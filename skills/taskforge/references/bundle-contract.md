# task-bundle.zip — what's in it

The deliverable. Simple and dual-use: hand a candidate `task/` and grade by hand with `EVALUATION.md`,
or send the whole bundle to heedful for automated grading.

```
task-bundle.zip
├── task/                    the exercise the candidate works on
│   ├── BRIEF.md             the only thing the candidate reads (problem, not solution)
│   ├── <source files>       a fix task ships its failing test; a build task ships no tests for it
│   └── package-lock.json    (or equivalent) — deps are installed by whoever runs it, NOT vendored here
├── EVALUATION.md            human + AI readable grading guide (problem, rubric, NOTES expectations,
│                            how to grade fix vs build)
├── evaluation/
│   └── reference/           the team's solution for the files the candidate builds/fixes — ONE
│                            acceptable approach, never a similarity target
└── context.json            app metadata (not grading): who made it, source PR/issue, discussion
                            summary, role/seniority, suitability verdict, language + run commands,
                            the machine-readable `rubric`, and the `install_command`
```

What it deliberately does **not** contain: no `node_modules`/vendored deps (the lockfile is enough;
heedful installs at ingest, a human runs `npm install`), no hidden tests, no scorecard blob, no manifest,
no checksums.

## context.json (app metadata)
```jsonc
{
  "task_id": "string",
  "created_at": "ISO-8601",
  "created_by": { "operator": "from git config", "gh_login": "...", "email": null },
  "source": { "repo": "owner/name",
              "pr": { "number": 0, "title": "", "url": "", "description": "" },
              "issue": { "number": 0, "title": "", "url": "", "body": "" } },
  "summary": "what the operator and the skill decided when designing this task",
  "assessment": { "problem_summary": "...", "test_focus": "...", "skills_assessed": ["..."] },
  "hiring": { "position": "...", "seniority": "senior", "job_description": "...", "time_target_hours": 1.5 },
  "pr_suitability": { "verdict": "recommend|flag|refuse", "reasons": ["..."] },
  "rubric": [ { "dimension": "...", "acceptable_approaches": ["..."], "what_good_looks_like": "..." } ],
  "task_mode": "descriptive string (e.g. 'build (senior): design the model + fix a bug')",
  "language": "node", "build_command": null, "install_command": "npm ci", "test_command": "npm test",
  "skill_version": "...", "spec_version": "agentskills.io",
  // OPTIONAL — frontend tasks only (both or neither; absent ⇒ backend task, nothing changes):
  "dev_command": "npm run dev", "preview_port": 3000
}
```

## Frontend tasks (`dev_command` + `preview_port`)

A task whose deliverable is a running web app declares its dev server:

- `dev_command` — the command the candidate runs to start the dev server. It is the **sole source
  of the BRIEF's run instruction** — the readme template renders it, so the field and what the
  candidate is told can't drift. `language` stays `"node"` — frontend-ness is task data, not a
  language fork.
- `preview_port` — the local port the dev server listens on. heedful injects it as
  `HEEDFUL_PREVIEW_PORT` and wires the in-IDE "Open preview" button and Ports-view links to
  `/absproxy/<preview_port>/`.

**Invariant (enforced by `scripts/validate_bundle.py`):** the baked basePath fallback in
`task/next.config.*` must be exactly `/absproxy/<preview_port>` (e.g.
`basePath: process.env.HEEDFUL_BASE_PATH ?? "/absproxy/3000"` with `preview_port: 3000`). If the
port and the baked path disagree, the preview button, the Ports links, and every root-relative
asset all break — and nothing else catches it. A frontend bundle also hard-requires a runnable
`test_command` (a build task's suite is the only automated check the template ships).

## EVALUATION.md (grading guide)
Generated by `package.py` from the task design: the problem, what's being assessed, how to grade (fix =
the shipped test should pass; build = judge the result against the rubric + NOTES, with
`evaluation/reference/` as one acceptable approach), the rubric dimensions, NOTES expectations, and any
planted bugs. Readable by a human grader and feedable to an eval agent.

## Safety
`package.py` scrubs `EVALUATION.md` + `context.json` prose for secrets (fail-closed — no zip if a real
secret is found) and redacts PII (committer emails). `node_modules`/vendored dirs are excluded from
`task/`. The reference solution is under `evaluation/reference/`, never under `task/`.
